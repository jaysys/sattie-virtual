# 개발계획

## 1. 목적
- `2-설계자/` 폴더의 설계 문서를 기준으로 MVP 개발 순서를 확정한다.
- 구현, 검증, 이슈 정리를 모두 `3-개발자/` 폴더 안에서 관리한다.
- `1-기획자/`, `2-설계자/` 폴더는 읽기 전용으로 유지한다.

## 2. 기준 문서
- `2-설계자/마크다운_뷰어_웹서비스_설계서.md`
- `2-설계자/API_응답_명세_부록.md`
- `2-설계자/화면별_와이어프레임_컴포넌트_설계서.md`
- `1-기획자/마크다운_뷰어_웹서비스_기획서.md`

## 3. 현재 구현 기준
- 서버는 FastAPI 기반이며 `app/` 아래에 API와 UI 코드가 존재한다.
- 현재 주요 구현 파일은 `app/core.py`, `app/sattie_api.py`, `app/md_viwer.py`, `app/ui/`다.
- 설계서 기준으로 문서 서비스 전용 라우트, 문서 인덱스 구조, 검색/업로드/운영 화면이 MVP 핵심 범위다.

## 4. 개발 원칙
- 설계 문서의 MVP 범위와 단계별 우선순위를 그대로 따른다.
- 개발 진행 중 생성되는 계획, 메모, 검증 기록은 모두 `3-개발자/`에 작성한다.
- `1-기획자/`, `2-설계자/` 문서는 읽기만 하고 수정하지 않는다.
- 설계와 실제 코드가 충돌하면 원문 대신 `3-개발자/`에 차이를 기록한다.

## 5. 단계별 개발 계획
### 1단계. 문서 서비스 기반 정리
- 문서 서비스 전용 라우트 구조를 분리한다.
- 문서 메타데이터, slug, front matter, 인덱스 구조를 설계서 기준으로 정리한다.
- 파일 시스템 + JSON 인덱스 기반 MVP 구조를 우선 구현한다.

### 2단계. 문서 API 구현
- `GET /api/docs/list`
- `GET /api/docs/search`
- `GET /api/docs/{slug}`
- `GET /api/docs/{slug}/raw`
- `POST /api/docs/upload`
- `POST /api/docs/reindex`
- API 응답 형식은 `2-설계자/API_응답_명세_부록.md`와 정합성을 맞춘다.

### 3단계. 화면 구현
- 랜딩 페이지
- 문서 허브 페이지
- 문서 상세 페이지
- 업로드 페이지
- 운영자 문서 관리 페이지
- 화면 구조와 컴포넌트 경계는 `2-설계자/화면별_와이어프레임_컴포넌트_설계서.md`를 기준으로 구현한다.

### 4단계. 품질 보강
- 검색 정확도와 정렬 규칙 보강
- SEO 메타 및 공유 카드 반영
- 렌더링 안전성, 업로드 검증, 에러 처리 보강
- 모바일 대응, 접근성, 회귀 테스트 수행

## 6. 산출물 계획
- `3-개발자/개발계획.md`: 기준 계획 문서
- `3-개발자/작업일지.md`: 구현 진행 이력
- `3-개발자/검증기록.md`: 테스트 및 검증 결과
- `3-개발자/이슈메모.md`: 설계 충돌, 리스크, 보완 필요사항

## 7. 완료 기준
- 설계서의 MVP 범위가 실제 라우트, API, UI로 구현되어야 한다.
- API 응답이 설계 부록과 일치해야 한다.
- 핵심 화면이 와이어프레임과 사용자 흐름 기준을 충족해야 한다.
- 개발 과정에서 생성된 문서는 모두 `3-개발자/` 폴더에 정리되어야 한다.
