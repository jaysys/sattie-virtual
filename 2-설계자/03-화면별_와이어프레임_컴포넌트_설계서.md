# 화면별 와이어프레임 컴포넌트 설계서

## 1. 문서 목적
- 마크다운 뷰어 웹서비스의 핵심 화면 구조를 와이어프레임 수준으로 고정한다.
- 구현팀이 레이아웃, 컴포넌트 경계, 사용자 흐름을 해석 없이 구현할 수 있도록 기준을 제공한다.

## 2. 화면 구성 원칙
- 문서 읽기 경험이 최우선이다.
- 탐색 요소는 풍부해야 하지만 본문을 방해하면 안 된다.
- 데스크톱은 3단 레이아웃, 모바일은 드로어/스택 구조를 기본으로 한다.
- 허브와 상세의 UI 톤은 일관되게 유지한다.

## 3. 화면 목록
- 랜딩 페이지
- 문서 허브 페이지
- 문서 상세 페이지
- 업로드 페이지
- 운영자 문서 관리 페이지

## 4. 랜딩 페이지 설계

### 4.1 목적
- 서비스 가치 전달
- 허브 유입
- 업로드 유도

### 4.2 와이어프레임
```text
+--------------------------------------------------------------+
| Logo | Hub | Upload | Search                                |
+--------------------------------------------------------------+
| Hero Title                                                   |
| "마크다운 파일을 올리면 바로 문서 사이트가 됩니다"            |
| [샘플 문서 보기] [문서 업로드]                               |
| Supporting copy                                              |
+--------------------------------------------------------------+
| Feature Cards                                                |
| [Mermaid] [MathJax] [SEO] [Search]                           |
+--------------------------------------------------------------+
| Sample Collections                                           |
| [API Docs] [Architecture] [Runbook]                          |
+--------------------------------------------------------------+
| Footer                                                       |
+--------------------------------------------------------------+
```

### 4.3 핵심 컴포넌트
- HeaderNav
- HeroSection
- PrimaryActionButtons
- FeatureCardGrid
- SampleCollectionGrid
- Footer

```text
| - Sort                   |                                        |

|                          | [List Item]                            |
|                          | [List Item]                            |
|                          |                                        |
|                          | Section: All Documents                 |
|                          | [Document Row/Card...]                 |

```

### 5.3 핵심 컴포넌트

- CategoryTabs

### 5.4 문서 카드 규격
- 제목
- 1줄 요약
- 카테고리
- 태그 2~3개
- 업데이트 일시
- 예상 읽기 시간
- 상세 이동 링크

## 6. 문서 상세 페이지 설계

### 6.1 목적
- 긴 문서를 편하게 읽게 한다.
- 현재 문서와 주변 문서를 함께 탐색하게 한다.
- 공유와 섹션 링크 복사를 쉽게 만든다.

### 6.2 데스크톱 와이어프레임
```text
+--------------------------------------------------------------------------------+
| Header: Logo | Search | Upload | Hub                                           |
+----------------------+--------------------------------+------------------------+
| Left Doc Sidebar     | Main Article                   | Right TOC Sidebar      |
| - Collection title   | Title                          | - H2                   |
| - Category list      | Summary                        | - H2                   |
| - Neighbor docs      | Meta row                       | - H3                   |
|                      | Tags                           | Share buttons          |
|                      | ----------------------------   | Related docs short     |
|                      | Rendered markdown body         |                        |
|                      |                                |                        |
+----------------------+--------------------------------+------------------------+
| Bottom Related Documents                                                     |
+--------------------------------------------------------------------------------+
```

### 6.3 모바일 와이어프레임
```text
+--------------------------------------+
| Header: Menu | Logo | Search         |
+--------------------------------------+
| Title                                |
| Summary                              |
| Meta row                             |
| [TOC] [Share] [Raw]                  |
| -----------------------------------  |
| Rendered markdown body               |
|                                      |
| Related documents                    |
+--------------------------------------+
```

### 6.4 핵심 컴포넌트
- ArticleHeader
- ArticleMetaRow
- TagList
- ArticleBody
- TocSidebar
- SharePanel
- RelatedDocuments
- RawMarkdownButton

### 6.5 상세 UX 규칙
- H2, H3 heading에는 링크 복사 아이콘을 제공한다.
- 코드블록에는 복사 버튼을 제공한다.
- 이미지 클릭 시 확대 모달을 제공한다.
- TOC는 현재 읽고 있는 섹션을 강조할 수 있어야 한다.
- 문서가 길어도 본문 가독성이 무너지지 않아야 한다.

## 7. 업로드 페이지 설계

### 7.1 목적
- 운영자 또는 사용자가 `.md` 파일을 빠르게 등록
- 업로드 후 즉시 상세 페이지 진입

### 7.2 와이어프레임
```text
+--------------------------------------------------------------+
| Header                                                       |
+--------------------------------------------------------------+
| Upload Title                                                 |
| [파일 선택]                                                  |
| Selected filename                                             |
| [업로드]                                                     |
| Result message                                                |
| Recent uploads(optional)                                      |
+--------------------------------------------------------------+
```

### 7.3 핵심 컴포넌트
- UploadPanel
- FilePicker
- UploadButton
- UploadStatusMessage
- UploadResultLink

### 7.4 업로드 UX 규칙
- `.md` 이외 파일은 선택 단계 또는 업로드 단계에서 즉시 차단한다.
- 업로드 중 상태를 명시한다.
- 완료 후 문서 URL을 바로 보여준다.
- 실패 시 원인과 재시도 행동이 함께 보여야 한다.

## 10. 상태별 화면 설계

### 10.1 빈 상태
- 문서가 없을 때
  - "첫 문서를 업로드하세요"
  - 업로드 버튼 우선 노출

### 10.2 검색 결과 없음
- 검색어 표시
- 필터 초기화 버튼
- 추천 카테고리 링크 제공

### 10.3 렌더링 실패
- 문서 자체는 존재하지만 렌더링 실패
- 원본 Markdown 다운로드 제공
- 운영자 문의 또는 재색인 안내 제공

### 10.4 업로드 실패
- 실패 사유 표시
- 허용 파일 형식과 용량 제한 재노출

## 11. 반응형 설계 원칙
- `>= 1280px`
  - 3단 레이아웃 사용
- `768px ~ 1279px`
  - 좌측 사이드바 유지, 우측 TOC는 축소 또는 접기
- `< 768px`
  - 본문 우선 단일 컬럼
  - TOC와 문서 탐색은 드로어 또는 바텀시트

## 12. 접근성 설계 기준
- 키보드만으로 검색, 이동, 업로드가 가능해야 한다.
- TOC, 공유 버튼, 업로드 버튼에는 명확한 라벨이 있어야 한다.
- heading 계층은 문서 구조와 일치해야 한다.
- 색상만으로 상태를 구분하지 않는다.

## 13. 화면 간 사용자 흐름

### 13.1 신규 사용자 흐름
1. 랜딩 진입
2. 샘플 문서 보기 또는 허브 진입
3. 문서 상세 열람
4. 공유 또는 다른 문서 탐색

### 13.2 업로드 사용자 흐름
1. 업로드 페이지 진입
2. `.md` 파일 선택
3. 업로드
4. 성공 메시지 확인
5. 문서 상세 페이지 이동

### 13.3 검색 중심 사용자 흐름
1. 허브 진입
2. 검색어 입력
3. 결과 목록 확인
4. 상세 진입
5. 관련 문서로 이동

## 14. 구현 우선순위
- 1순위
  - 문서 허브
  - 문서 상세
  - 업로드 페이지
- 2순위
  - 운영자 문서 관리 페이지
  - 빈 상태, 에러 상태, 검색 결과 없음 상태
- 3순위
  - 시각적 고도화
  - 추천 영역 고도화

## 15. 최종 기준
- 화면 설계는 본문 읽기 경험을 해치지 않아야 한다.
- 허브와 상세는 서로 유기적으로 연결되어야 한다.
- 업로드 후 즉시 열람이라는 핵심 가치가 모든 화면에서 드러나야 한다.
