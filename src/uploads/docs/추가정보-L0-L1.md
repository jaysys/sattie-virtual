### CADU L0 L1

위성 데이터 처리 과정에서 **CADU**와 **L0**는 데이터의 **'가공 전 상태'**와 **'가공 후 첫 번째 단위'**를 구분하는 가장 중요한 기준입니다. 이 둘의 차이를 명확히 이해하면 데이터 파이프라인의 시작점을 정확히 파악할 수 있습니다.

### 1. CADU (Channel Access Data Unit) : 물리 계층의 원시 데이터

위성이 지상국 안테나를 향해 쏟아내는 **물리적 신호** 그 자체입니다.

- **상태:** 데이터가 위성에서 전송될 때 오류 방지를 위해 프레임 단위로 묶인 **바이너리 비트 스트림(Bit Stream)**입니다.
- **특징:** \* 암호화되어 있거나, 위성 통신 규격(예: CCSDS 표준)에 맞춰 패킷화되어 있습니다.
- 아직 영상이라고 부르기 어려운 '통신용 데이터 덩어리'입니다.

- **저장 형식:** 주로 `.dat` 또는 `.ccsds` 확장자를 가집니다.
- **역할:** 지상국 안테나가 수신한 무선 전파를 디지털 파일로 저장한 최하위 데이터입니다.

---

### 2. L0 (Level 0) 영상 : 논리 계층의 원시 영상

수신된 CADU에서 **'위성 통신용 패킷을 걷어내고, 실제 영상 데이터만 추출'**하여 정리한 단계입니다.

- **상태:** CADU 내부에 들어있던 복잡한 통신 헤더를 제거(복호화/디코딩)하고, 영상 센서의 픽셀 정보를 순서대로 재구성한 상태입니다.
- **특징:**
- 아직 기하학적 보정(좌표 입히기)이 되지 않은 '원본 픽셀 값'입니다.
- 그러나 분석을 위해 프레임(Frame) 또는 패킷 단위로 깔끔하게 정리되어 있습니다.

- **저장 형식:** `*.raw`, `*.L0F`, `*.idx`, `*.ibmu` 등이 포함된 파일 세트.
- **역할:** 정밀 분석(L1)을 위해 즉시 로딩 가능한 상태의 영상 기초 자료입니다.

---

### 3. 비교 요약표

| 구분            | CADU                           | L0 (Level 0)                     |
| --------------- | ------------------------------ | -------------------------------- |
| **정의**        | 통신 계층의 프레임 단위 데이터 | 논리 계층의 복호화된 영상 데이터 |
| **상태**        | 암호화/패킷화된 원시 신호      | 패킷 제거 후 영상 재구성 완료    |
| **주요 목적**   | 데이터 전송 및 수신 확인       | 정밀 분석(L1)을 위한 입력값      |
| **분석 가능성** | 불가 (통신 프로토콜 이해 필요) | 가능 (단, 기하 보정 전 상태)     |

---

### 핵심 요약: 왜 나누어 관리하나요?

시스템이 **CADU에서 L0로 넘어갈 때(복호화 단계)** 가장 큰 연산 부하가 발생합니다.

- 위성마다, 촬영 모드마다 통신 규격이 다를 수 있기 때문에 **CADU는 '보관용'**으로,
- 이를 표준화된 **L0로 변환한 데이터는 '분석용'**으로 분리하여,
- 만약 분석 과정에서 오류가 나도 **CADU(Raw 데이터)만 있으면 언제든 L0부터 다시 뽑아낼 수 있도록 설계**하기 위함입니다.

이 두 단계가 시스템의 **데이터 무결성(Data Integrity)**을 지키는 방어선입니다.

---

### L0 데이터 구성요소별 상세 역할

L0(Level 0) 단계에서 언급하신 다양한 파일 확장자들은 위성 데이터의 **무결성(Integrity)**과 **구조적 정보**를 유지하기 위해 분할되어 관리되는 구성요소들입니다. 각 파일의 역할은 시스템이 원시 데이터로부터 유의미한 정보를 재구성하는 데 필수적입니다.

각 파일은 위성 데이터의 **'뼈와 살'**을 구성한다고 보시면 됩니다.

| 파일 확장자  | 역할 및 설명                                                                                                                                          |
| ------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`*.raw`**  | **영상 데이터 본체.** 위성 센서가 촬영한 실제 비트(bit) 값이 담긴 바이너리 데이터로, 가장 큰 용량을 차지합니다.                                       |
| **`*.ceu`**  | **위성 제어 정보(Control/Configuration).** 위성의 센서 설정 값, 촬영 당시의 모드 정보를 담고 있습니다.                                                |
| **`*.idx`**  | **인덱스 파일.** `*.raw` 데이터 내의 각 패킷(Packet)이나 프레임(Frame)이 파일의 어느 위치(Offset)에 있는지 알려주는 일종의 '목차'입니다.              |
| **`*.ibmu`** | **위성 자세/궤도 정보.** 촬영 당시 위성이 어느 위치에서 어떤 방향을 보고 있었는지에 대한 보조 데이터입니다. 영상 정밀 보정에 필수적입니다.            |
| **`*.L0F`**  | **L0 포맷 정의/헤더.** 이 데이터 세트 전체를 정의하는 구조체 파일입니다. 전체 영상의 크기, 촬영 시간, 필터 설정 등 최상위 메타데이터를 담고 있습니다. |
| **`*.jpg`**  | **퀵룩(Quick-Look).** 분석관이 전체 `*.raw`를 열어보기 전, 데이터의 상태와 촬영 범위를 즉시 확인할 수 있게 해주는 저해상도 미리보기 이미지입니다.     |

### 왜 이렇게 파일을 나누어 관리하나요?

1. **데이터 병렬 처리:** 시스템이 `*.idx`를 통해 데이터의 위치를 파악하면, 22대의 처리 서버가 원본 데이터를 처음부터 읽지 않고도 필요한 부분만 즉시 찾아내어 병렬로 작업할 수 있습니다.
2. **메타데이터 분리:** 영상 데이터(raw)를 변형시키지 않고도, 촬영 조건(ceu)이나 위성 궤도 정보(ibmu)를 독립적으로 관리하고 수정할 수 있어 추후 재처리(Reprocessing)가 용이합니다.
3. **검증(Validation):** 시스템은 `*.L0F`와 인덱스 정보를 활용해 수신된 데이터가 누락 없이 완벽하게 들어왔는지(Data Integrity)를 자동으로 검증합니다.

### 핵심 프로세스 요약

이 단계에서 **L0F**와 **idx**는 시스템의 '지도' 역할을 하며, **raw**와 **ibmu**는 '재료' 역할을 합니다. 처리 서버는 이 재료들을 조합하여 기하학적/방사학적 보정을 거친 **L1(분석용)** 영상을 생성하게 됩니다.

---

### L1 파일 구성요소의 역할

L1(Level 1) 단계는 L0의 원시 데이터를 기하학적·방사학적으로 보정하여 **실제 지도 위에서 위치 정보를 가진 '분석 가능한 영상'**으로 변환하는 최종 처리 단계입니다. 이 파일들은 단순한 사진을 넘어 **GIS(지리정보시스템)와 완벽하게 연동**되는 구조를 갖추고 있습니다.

각 파일은 위성 영상의 **위치 정확도**와 **분석 가능성**을 결정짓는 핵심 데이터들입니다.

| 파일 확장자               | 역할 및 설명                                                                                                                                              |
| ------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`*.tiff` (GeoTIFF)**    | **영상 본체.** 지리적 좌표 정보가 내장된 고해상도 이미지 파일입니다.                                                                                      |
| **`*.jgw`**               | **World File.** TIFF 파일의 각 픽셀이 지구상의 어느 위도/경도와 일치하는지 알려주는 좌표 변환 계수를 담은 텍스트 파일입니다.                              |
| `*.shp`, `*.dbf`, `*.shx` | **벡터 레이어 세트 (Shapefile).** 영상 내 특정 지역, 건물, 표적의 경계선을 나타내는 벡터 데이터입니다. 분석관이 변화탐지를 위해 사용하는 핵심 파일입니다. |
| **`*.xml`**               | **메타데이터.** 촬영 날짜, 센서 파라미터, 보정 히스토리 등이 기록된 문서로, 데이터 관리 및 시스템 연동의 핵심입니다.                                      |
| **`*.txt`**               | **품질 보고서.** 처리 과정에서 발생한 오차 범위나 구름 양 등을 기록한 텍스트 파일입니다.                                                                  |

---

### 왜 이 파일들이 함께 존재해야 하나요?

1. **공간 정보의 정밀성 (Geo-referencing):** `*.tiff` 파일만 있으면 단순한 그림에 불과하지만, `*.jgw`나 `*.xml`이 결합되어야만 이 영상이 지구의 정확히 어디를 가리키는지 알 수 있습니다.
2. **분석의 자동화:** 변화탐지 시스템은 `*.shp` 파일을 이용해 특정 건물의 경계선을 딴 뒤, 시계열 데이터와 비교합니다. 즉, **L1은 '보는 영상'에서 '측정하는 데이터'로 변모한 상태**입니다.
3. **범용성:** 이 파일 구성은 OGC(Open Geospatial Consortium) 등 국제 표준을 준수하여, 별도의 특수 소프트웨어 없이도 범용 GIS 도구(QGIS, ArcGIS 등)에서 즉시 불러와 중첩 분석이 가능합니다.

### 데이터 처리 흐름 요약

- **LP/SARP 처리:** 위성 센서의 기울어짐, 지구의 곡률, 대기 산란 등으로 인한 왜곡을 수학적 모델($$Polynomial \ or \ Thin \ Plate \ Spline$$ 등)을 사용하여 바로잡습니다.

- **최종 결과물:** 시스템은 처리 완료된 `*.tiff`와 벡터 파일(`*.shp`)을 영상분석 서버로 전송하며, 분석관은 이를 통해 변화탐지, 표적 식별, 주제도 생성 작업을 수행합니다.

**관리자나 분석관 입장에서 L1 데이터의 좌표 정확도(GCP, Ground Control Point) 관리**가 시스템 전체의 판독 신뢰도를 좌우한다고 볼 수 있습니다.
