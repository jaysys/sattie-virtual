# 개발 인수인계 명세서

## 1. 문서 목적
- 설계 패키지를 실제 개발 작업 단위로 분해한다.
- 구현팀이 우선순위, 선후관계, 완료 기준을 해석 없이 이해하도록 한다.

## 2. 기준 문서
- `2-설계자/마크다운_뷰어_웹서비스_설계서.md`
- `2-설계자/API_응답_명세_부록.md`
- `2-설계자/화면별_와이어프레임_컴포넌트_설계서.md`

## 3. 개발 목표
- 현재 FastAPI 애플리케이션 안에 문서 허브 기능을 독립 서비스 수준으로 추가한다.
- 기존 `/guide`와 `/ui` 기능을 깨지 않으면서, `/hub`, `/p/{slug}`, `/api/docs/*` 흐름을 구현한다.

## 4. 작업 원칙
- 큰 구조 변경보다 기능 단위 점진 구현을 우선한다.
- 읽기 기능을 먼저 안정화하고, 쓰기 기능은 그 다음에 붙인다.
- UI와 API는 동시에 만들되, 인덱서와 라우트부터 선행한다.

## 5. 작업 분해

### 5.1 Epic A: 문서 서비스 기본 골격

#### Task A-1. 문서 서비스 모듈 구조 추가
- 목적
  - 문서 서비스 로직을 기존 위성영상 데모 로직과 분리
- 구현 범위
  - `routes`
  - `service`
  - `indexer`
  - `models`
  - `seo`
- 완료 기준
  - 문서 서비스 모듈이 독립 import 가능
  - 기존 API와 충돌 없음

#### Task A-2. 라우트 연결
- 목적
  - `/`, `/hub`, `/p/{slug}`, `/upload`, `/api/docs/*` 기본 라우트 연결
- 선행 조건
  - Task A-1 완료
- 완료 기준
  - 각 경로가 기본 응답 또는 플레이스홀더라도 정상 응답

### 5.2 Epic B: 문서 인덱스 및 메타데이터

#### Task B-1. Markdown 스캐너 구현
- 목적
  - `guide/`, `docs/`, `uploads/docs/`를 스캔
- 구현 범위
  - `.md` 탐색
  - 허용 경로 규칙
  - 경로 정규화
- 완료 기준
  - 대상 문서 목록을 안정적으로 반환

#### Task B-2. front matter 파서 및 기본 메타 추출기 구현
- 목적
  - `title`, `summary`, `category`, `tags`, `is_public`, `order` 추출
- 선행 조건
  - Task B-1 완료
- 완료 기준
  - front matter가 있어도, 없어도 메타가 생성됨

#### Task B-3. slug 생성기 구현
- 목적
  - 파일명 기반 slug 생성과 충돌 처리
- 완료 기준
  - 같은 제목 또는 유사 파일명이 있어도 slug 충돌이 해결됨

#### Task B-4. heading/목차 추출기 구현
- 목적
  - H2~H3 중심 목차 데이터 생성
- 완료 기준
  - 상세 페이지 TOC에 바로 쓸 수 있는 배열이 생성됨

#### Task B-5. 문서 인덱스 JSON 생성기 구현
- 목적
  - `documents.json`, `search.json` 생성
- 선행 조건
  - Task B-1 ~ B-4 완료
- 완료 기준
  - 전체 문서 인덱스가 재생성 가능
  - 업로드 시 단건 반영 또는 전체 재생성 중 하나의 일관된 전략을 유지
  - 기존 인덱스 fallback 전략 유지

### 5.3 Epic C: 문서 렌더링 및 상세 페이지

#### Task C-1. Markdown 상세 렌더링 조합기 구현
- 목적
  - 원문 + 메타 + HTML + headings를 합친 상세 응답 생성
- 선행 조건
  - Epic B 완료
- 완료 기준
  - API와 HTML 양쪽에서 사용할 수 있는 상세 객체 생성

#### Task C-2. heading anchor 주입
- 목적
  - 본문 헤더에 직접 링크 가능 anchor 부여
- 완료 기준
  - TOC 클릭 시 해당 섹션 이동 가능

#### Task C-3. 관련 문서 추천 로직
- 목적
  - 동일 카테고리 우선 관련 문서 3~5개 제공
- 완료 기준
  - 상세 API 또는 상세 페이지에서 관련 문서 노출 가능

### 5.4 Epic D: JSON API

#### Task D-1. 목록 API
- 경로
  - `GET /api/docs/list`
- 완료 기준
  - category, tag, sort, page, page_size 처리

#### Task D-2. 검색 API
- 경로
  - `GET /api/docs/search`
- 완료 기준
  - title, tags, summary, headings, body 일부 검색 가능

#### Task D-3. 상세 API
- 경로
  - `GET /api/docs/{slug}`
- 완료 기준
  - html, headings, related_documents 포함

#### Task D-4. 원문 API
- 경로
  - `GET /api/docs/{slug}/raw`
- 완료 기준
  - 원본 Markdown 반환

#### Task D-5. 업로드 API
- 경로
  - `POST /api/docs/upload`
- 완료 기준
  - 저장 후 재색인, slug, public_url 반환

#### Task D-6. 재색인 API
- 경로
  - `POST /api/docs/reindex`
- 완료 기준
  - 전체 재색인 및 문서 수 반환

#### Task D-7. 운영자 접근 제어
- 목적
  - 공개 사용자 화면과 내부 운영 기능을 분리
- 완료 기준
  - `/admin/docs`와 `POST /api/docs/reindex`는 최소한 내부 운영 경로로만 노출
  - MVP에서는 로그인 대신 환경변수 기반 간단 보호 또는 내부망 운영 방식을 문서화

### 5.5 Epic E: HTML 화면

#### Task E-1. 랜딩 페이지
- 완료 기준
  - 허브 진입, 업로드 유도, 서비스 가치 제시 가능

#### Task E-2. 문서 허브 페이지
- 완료 기준
  - 문서 카드 목록, 카테고리, 검색, 정렬 노출

#### Task E-3. 문서 상세 페이지
- 완료 기준
  - 본문, 메타, TOC, 관련 문서, 공유, 원문 링크 표시

#### Task E-4. 업로드 페이지
- 완료 기준
  - 파일 선택, 업로드 상태, 완료 후 상세 이동 가능

#### Task E-5. 운영자 문서 관리 페이지
- 완료 기준
  - 문서 수, 공개 상태, 재색인 기능 노출

### 5.6 Epic F: 검색 품질 및 운영 안정화

#### Task F-1. 검색 점수 규칙 반영

#### Task F-2. 에러 처리 통일

#### Task F-3. 캐시와 재색인 안정화

## 7. 병렬 진행 가능 영역
- 병렬 가능
  - Epic B 일부와 Epic E 화면 마크업 초안
  - Epic D API 응답 모델과 Epic E 화면 컴포넌트 설계 반영
- 병렬 비권장
  - 인덱서 완성 전 검색 API 구현
  - 상세 객체 고정 전 상세 페이지 완성

## 8. 완료 체크리스트
- `/hub`가 열려야 한다.
- `/p/{slug}`가 실제 문서를 보여야 한다.
- `/api/docs/list`와 `/api/docs/search`가 정상 동작해야 한다.
- 업로드 후 바로 상세 페이지로 이동할 수 있어야 한다.
- `/guide`와 `/ui`에 회귀가 없어야 한다.

## 9. 인수인계 메모
- 메타데이터 구조와 응답 형식은 부록 문서를 우선 기준으로 삼는다.
- 화면 레이아웃과 컴포넌트 경계는 화면 설계서를 우선 기준으로 삼는다.
- 운영자 기능은 MVP에서 읽기 전용 관리 화면 + 재색인으로 제한한다.
- 구현 중 설계 충돌이 보이면 메인 설계서를 수정 요청 대상으로 보고, 코드에서 임의 판단하지 않는다.
