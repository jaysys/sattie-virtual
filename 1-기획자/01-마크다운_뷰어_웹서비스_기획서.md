# 마크다운 뷰어 웹서비스 기획서

## 1. 문서 목적
- 현재 폴더의 소스를 분석해, 기존 기능을 확장한 마크다운 문서 뷰어 웹서비스 론칭 전략을 제안한다.
- 목표는 단순한 "파일 렌더러"가 아니라 사용자가 반복 방문하고 공유하고 업로드하게 만드는 성장형 문서 플랫폼을 설계하는 것이다.

## 2. 현재 소스 기준 분석

### 2.1 확인된 현재 자산
- `app/sattie_api.py`
  - `guide/` 폴더 내 `.md` 파일 목록 조회 기능 제공
  - `/guide/{path}` HTML 렌더링 기능 제공
  - 웹 업로드로 `.md` 파일 저장 가능
- `app/md_viwer.py`
  - 경량 마크다운 렌더러 구현
  - 헤더, 리스트, 코드블록, 표, 링크, 이미지, 인라인 코드 처리
  - MathJax, Mermaid, 이미지 확대 모달 지원
- `guide/README.md`
  - 실제 문서 폴더가 이미 존재하며 서비스 시연에 바로 활용 가능

### 2.2 현재 서비스로서의 강점
- 이미 동작 가능한 마크다운 뷰어 핵심 기능이 있다.
- 서버 사이드 렌더링 형태로 빠르게 결과를 보여줄 수 있다.
- 별도 무거운 프론트 프레임워크 없이 빠르게 배포 가능하다.
- Mermaid, 수식, 이미지 확대까지 지원해 기술문서 친화적이다.
- 업로드 기능이 있어 초기 운영자가 직접 콘텐츠를 넣기 쉽다.

### 2.3 현재 한계
- 서비스 이름, 브랜드, 랜딩 구조가 없다.
- 문서 목록이 단순 링크 수준이라 탐색성이 약하다.
- 검색, 태그, 즐겨찾기, 최근 본 문서, 공유 카드가 없다.
- SEO 구조가 부족해 외부 유입을 만들기 어렵다.
- 사용자 계정, 협업, 권한, 문서 버전 관리가 없다.
- 업로드는 가능하지만 "왜 이 서비스를 계속 써야 하는지"에 대한 차별화 포인트가 약하다.

## 3. 서비스 방향 제안

### 3.1 서비스 한 줄 정의
"개발자와 실무자가 마크다운 문서를 가장 빠르게 읽고, 공유하고, 구조화해서 다시 찾을 수 있는 웹 문서 허브"

### 3.2 권장 포지셔닝
- 단순 뷰어 경쟁은 피한다.
- "기술문서 공유 특화", "수식/다이어그램 친화", "폴더 기반 즉시 문서 사이트화"를 전면에 둔다.
- 현재 코드베이스 특성상 다음 포지션이 가장 현실적이다.
  - 로컬 폴더를 바로 문서 사이트로 공개하는 서비스
  - 팀 문서/가이드/운영런북 전용 공개 뷰어
  - 개발자 포트폴리오형 마크다운 허브

## 4. 타겟 사용자

### 4.1 1차 타겟
- 개발자
- 스타트업 PM/기획자
- 기술영업, 솔루션 아키텍트
- 운영/인프라 문서를 자주 보는 팀

### 4.2 2차 타겟
- 대학 연구실
- AI/데이터 분석가
- 기술 블로그 대신 문서 중심으로 정리하는 1인 창작자

### 4.3 타겟이 원하는 가치
- Git 없이도 문서를 웹에 올리고 싶다.
- 수식과 다이어그램이 깨지지 않아야 한다.
- 검색과 탐색이 편해야 한다.
- 링크 하나로 팀이나 고객에게 보여주고 싶다.
- Notion보다 가볍고, GitHub README보다 보기 좋아야 한다.

## 5. 많은 사용자를 끌어들이는 핵심 전략

### 5.1 첫 번째 전략: "즉시 공개" 경험
- 폴더 또는 파일 업로드 후 즉시 문서 사이트가 생성되게 한다.
- 사용자 입장에서 가장 강력한 첫 경험은 "5초 안에 내 문서가 웹페이지가 되는 것"이다.
- 현재 `guide` 업로드 구조를 확장하면 MVP 구현 속도가 빠르다.

### 5.2 두 번째 전략: 검색 유입형 구조
- 각 문서를 고유 URL로 제공한다.
- 문서 제목, 설명, 헤더 구조를 메타태그로 반영한다.
- 공개 문서는 검색엔진에 노출되게 하여 롱테일 유입을 만든다.
- "Markdown Viewer"가 아니라 "기술문서 웹호스팅" 관점으로 SEO를 잡아야 한다.

### 5.3 세 번째 전략: 공유가 쉬운 문서 카드
- 문서 공유 시 제목, 요약, 커버 이미지를 자동 생성한다.
- Slack, KakaoTalk, LinkedIn, X에서 보기 좋은 OG 카드가 중요하다.
- 문서를 보는 사용자가 다시 다른 문서를 탐색하게 만드는 내부 링크 추천도 필요하다.

### 5.4 네 번째 전략: 기술문서 특화 기능
- Mermaid, MathJax, 코드블록 복사, 목차 자동 생성, 헤더 링크 복사
- 현재 코드에 이미 일부 기반이 있으므로 빠르게 차별화 가능하다.
- 기술문서 독자는 일반 블로그 독자보다 재방문율이 높다.

### 5.5 다섯 번째 전략: 업로드가 아니라 "지식 허브" 경험
- 문서 목록만 보여주지 말고 컬렉션, 카테고리, 시리즈, 추천문서 영역을 제공한다.
- 예:
  - API 가이드
  - 운영 매뉴얼
  - 아키텍처 문서
  - 튜토리얼 시리즈
- 사용자는 한 문서 때문에 들어와도 구조가 좋으면 여러 문서를 본다.

## 6. 제품 차별화 포인트

### 6.1 경쟁 대안과 비교
- GitHub README: 보기와 탐색성이 약함
- Notion 공개페이지: 무겁고 기술문서 렌더링 품질이 제한적
- 단순 Markdown Viewer: 공유성과 재방문성이 약함

### 6.2 본 서비스의 차별화 메시지
- 폴더 업로드만으로 문서 사이트 생성
- 수식, Mermaid, 이미지, 표를 안정적으로 렌더링
- 검색과 목차 중심의 읽기 경험
- 공개 문서 SEO 최적화
- 팀 운영문서와 개발문서에 최적화

## 7. 제품 기능 기획

### 7.1 MVP
- 마크다운 파일 목록
- 문서 렌더링
- 파일 업로드
- 이미지 확대
- Mermaid 렌더링
- 수식 렌더링
- 문서별 고유 URL
- 기본 랜딩 페이지

### 7.2 출시 1차 개선
- 문서 제목 자동 추출
- 좌측 문서 탐색 메뉴
- 우측 목차
- 문서 내 검색
- 전체 문서 검색
- 최근 업로드/인기 문서
- 태그/카테고리
- 코드블록 복사 버튼
- 다크모드

### 7.3 출시 2차 개선
- 사용자 로그인
- 개인 문서함
- 공개/비공개 전환
- 팀 워크스페이스
- 문서 버전 히스토리
- GitHub 저장소 연동 자동 동기화
- 커스텀 도메인

## 8. 추천 핵심 화면

### 8.1 메인 랜딩
- "파일 하나로 문서 사이트 만들기" 메시지 강조
- 업로드 버튼과 샘플 문서 보기 버튼 제공
- 기술문서 예시 템플릿 노출

### 8.2 문서 허브 홈
- 인기 문서
- 최근 업데이트
- 카테고리별 문서 묶음
- 검색창

### 8.3 문서 상세
- 좌측 문서 목록
- 중앙 렌더링 본문
- 우측 목차
- 상단 공유, 복사, 원문 다운로드
- 하단 관련 문서 추천

## 9. 유입 전략

### 9.1 SEO
- 공개 문서를 인덱싱 허용
- 헤더 기반 앵커 URL 제공
- 문서별 메타 설명 자동 생성
- `how-to`, `guide`, `tutorial`, `api`, `runbook` 키워드 기반 페이지 구조 설계

### 9.2 콘텐츠 시드 전략
- 초기에는 운영자가 직접 좋은 샘플 문서를 올려야 한다.
- 현재 `docs/`, `guide/`, `통합설계.md`를 기반으로 데모 허브를 구성할 수 있다.
- "실제 프로젝트 문서가 이렇게 보인다"는 신뢰를 먼저 줘야 한다.

### 9.3 공유 확산 전략
- 문서 링크 복사
- 문단별 링크 복사
- SNS 공유 카드
- 읽기 좋은 URL slug 적용

### 9.4 개발자 커뮤니티 유입
- Product Hunt, GitHub, Velog, 브런치, 개발자 커뮤니티에 배포
- "README를 바로 문서 사이트로 바꿔주는 서비스" 메시지가 반응성이 높다.

## 10. 수익화 방향

### 10.1 무료 플랜
- 공개 문서 수 제한
- 기본 업로드 용량 제한
- 기본 테마 제공

### 10.2 유료 플랜
- 비공개 문서
- 팀 협업
- 커스텀 도메인
- 브랜딩 제거
- 고급 분석
- Git 연동 자동 배포

### 10.3 광고 전략
- 현재 `app/ui/index.html`에는 광고 스크립트가 일부 들어가 있으나, 문서 서비스 초기에 광고 중심 전략은 비추천이다.
- 초반에는 광고보다 사용자 경험과 공유 확산을 우선해야 한다.

## 11. 현재 코드 기반 구현 우선순위

### 11.1 바로 활용 가능한 것
- FastAPI 기반 라우팅
- HTML 응답 구조
- 마크다운 렌더러
- Mermaid/MathJax 지원
- 업로드 기능

### 11.2 우선 개발해야 할 것
- 서비스 전용 랜딩 페이지
- 문서 메타데이터 추출기
- 목차 생성
- 전체 검색 인덱스
- 카테고리/태그 체계
- 읽기용 UI 개선

### 11.3 구조적으로 분리할 것
- 위성영상 데모 UI(`/ui`)와 문서 서비스 UI는 분리 운영 권장
- `/guide`는 내부 데모가 아니라 정식 서비스 경로로 재설계 필요
- 예시 권장 경로:
  - `/docs`
  - `/hub`
  - `/p/{slug}`

## 12. 단계별 실행 계획

### 12.1 1단계: 2주 MVP
- `guide`를 서비스 홈으로 개선
- 문서 목록 UI 개편
- 문서 상세 UI 개편
- 목차/검색/공유 기능 추가
- 샘플 문서 컬렉션 구성

### 12.2 2단계: 4주 베타
- 사용자별 문서 공간
- 공개 URL 발급
- SEO 메타 정비
- 인기 문서 집계
- 카테고리/태그 도입

### 12.3 3단계: 6~8주 정식 론칭
- GitHub 연동
- 팀 워크스페이스
- 비공개 문서
- 구독 플랜

## 13. KPI 제안
- 방문자 대비 문서 열람률
- 문서 업로드 완료율
- 공유 링크 생성률
- 재방문율(7일, 30일)
- 문서당 평균 체류시간
- 검색 사용률
- 공개 문서의 검색엔진 유입 비중

## 14. 최종 제안
- 이 프로젝트는 새로 만들 필요가 없다.
- 이미 존재하는 `guide` 렌더링 기능을 중심으로, "문서 사이트 생성 서비스"로 재포지셔닝하는 것이 가장 빠르고 현실적이다.
- 많은 사용자를 끌어들이려면 뷰어 기능보다 아래 3가지를 먼저 완성해야 한다.
  - 업로드 즉시 웹 문서화
  - 검색과 탐색 중심 UI
  - 공유와 SEO 중심 공개 구조

## 15. 권장 산출물
- 서비스명 후보: `Sattie Docs`, `MD Hub`, `ReadFast`, `DocDrop`
- 첫 론칭 메시지:
  - "마크다운 파일을 올리면 바로 문서 사이트가 됩니다."
  - "README, 운영가이드, API문서를 가장 읽기 좋게 보여주는 웹서비스"

## 16. 부록: 현재 소스에서 서비스 기회로 해석되는 포인트
- `guide` 폴더 구조는 곧 문서 컬렉션 구조다.
- 업로드 기능은 초기 CMS 역할을 한다.
- Mermaid/MathJax 지원은 개발자 대상 차별화 요소다.
- 현재 부족한 것은 렌더링 엔진이 아니라 사용자 유입과 탐색 설계다.
