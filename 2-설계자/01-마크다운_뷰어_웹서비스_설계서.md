# 마크다운 뷰어 웹서비스 설계서

## 1. 문서 목적
- `1-기획자/마크다운_뷰어_웹서비스_기획서.md`를 실제 구현 가능한 서비스 설계로 구체화한다.
- 현재 코드베이스를 최대한 재사용하면서, 마크다운 뷰어를 독립 서비스로 전환하는 구조를 정의한다.
- 설계 검토 결과와 구현 우선순위를 함께 정리한다.

## 1.1 설계 목표
- 구현팀이 추가 해석 없이 바로 개발을 시작할 수 있어야 한다.
- 현재 FastAPI 기반 구조를 유지하면서 문서 서비스를 독립 기능으로 분리해야 한다.
- 초기 론칭은 빠르게, 이후 확장은 무리 없이 가능해야 한다.

## 1.2 비범위
- 이번 설계 단계에서 모바일 앱은 다루지 않는다.
- 실시간 공동편집 기능은 다루지 않는다.
- 외부 결제 시스템과 구독 청구 상세 설계는 다루지 않는다.
- 다국어 번역 파이프라인은 MVP 범위에서 제외한다.

## 1.3 관련 부속 문서
- `2-설계자/API_응답_명세_부록.md`
  - JSON API 응답 형식과 상태코드 상세 명세
- `2-설계자/화면별_와이어프레임_컴포넌트_설계서.md`
  - 화면 구조, 와이어프레임, 컴포넌트 경계 설계
- `2-설계자/README.md`
  - 설계 문서 패키지 개요와 권장 읽기 순서

## 2. 기준 소스와 현황

### 2.1 기준 소스
- `app/sattie_api.py`
- `app/md_viwer.py`
- `guide/`
- `docs/`

### 2.2 현재 구현 상태 요약
- 서버는 FastAPI 기반이다.
- `guide/` 폴더의 `.md` 문서를 목록 조회하고 HTML로 렌더링할 수 있다.
- `.md` 파일 업로드 후 `guide/`에 저장할 수 있다.
- Mermaid, MathJax, 이미지 링크, 표, 코드블록 처리가 가능하다.

### 2.3 현재 구조의 문제
- 루트 서비스는 위성영상 데모가 중심이며 문서 서비스가 보조 기능으로 붙어 있다.
- `/docs` 경로는 FastAPI Swagger와 충돌하므로 문서 서비스 메인 경로로 쓰면 안 된다.
- 문서 메타데이터 계층이 없어 검색, 정렬, 추천, SEO 최적화가 어렵다.
- 렌더링 결과는 있으나 서비스 탐색 UI가 거의 없다.

## 3. 설계 검토 결과

### 3.1 유지할 요소
- FastAPI 라우팅 구조
- 서버 사이드 HTML 렌더링 방식
- `guide/` 폴더 기반 문서 소스 구조
- `app/md_viwer.py`의 경량 렌더러
- Mermaid/MathJax 지원

### 3.2 변경할 요소
- 문서 서비스 경로 분리
- 문서 메타데이터 인덱스 추가
- 문서 목록/상세 UI 재설계
- 업로드를 운영 도구 수준에서 서비스 기능 수준으로 승격
- 문서 탐색, 검색, 공유, SEO 대응 추가

### 3.3 핵심 설계 판단
- 기존 `/guide`는 내부 운영용 원형 기능으로 유지 가능하다.
- 정식 사용자 서비스는 `/hub`, `/p/{slug}`, `/api/docs/*` 계열로 분리한다.
- Swagger 기본 경로 `/docs`는 유지하고, 서비스 문서 경로로 사용하지 않는다.
- MVP 단계에서는 DB 없이 파일 시스템 + JSON 인덱스로 시작하는 것이 가장 빠르다.

## 3.4 MVP 범위 고정
- 아래 항목은 MVP에 반드시 포함한다.
  - 문서 허브 메인
  - slug 기반 문서 상세 페이지
  - Markdown 업로드
  - 문서 메타데이터 인덱스 생성
  - 전체 문서 검색
  - 목차, 공유 메타, 헤더 앵커
- 아래 항목은 MVP에서 제외한다.
  - 로그인
  - 팀 협업
  - 비공개 문서
  - GitHub 자동 동기화
  - 커스텀 도메인

### 4.3 구조 분리 원칙
- 위성영상 데모 로직과 문서 서비스 로직은 파일 수준에서 분리한다.
- `sattie_api.py`는 최종적으로 조립 레이어 역할만 하게 만든다.
- Markdown 렌더링 로직은 유지하되, 메타 추출과 목차 생성을 분리한다.

### 4.4 컴포넌트 책임 분리
- `routes.py`
  - HTTP 요청/응답, 파라미터 검증, 템플릿 결합 담당
- `service.py`
  - 문서 조회, 목록 구성, 검색 호출, 상세 조합 담당
- `indexer.py`
  - 파일 스캔, front matter 해석, slug 생성, search index 생성 담당
- `models.py`
  - 문서 메타데이터와 검색 응답 스키마 담당
- `seo.py`
  - title, description, canonical, OG 메타 조립 담당

### 5.1 주요 화면
- `/`
  - 서비스 랜딩 페이지
- `/hub`
  - 문서 허브 메인
- `/hub/category/{category}`
  - 카테고리별 목록
- `/p/{slug}`
  - 문서 상세 페이지
- `/upload`
  - 업로드 화면
- `/admin/docs`
  - 운영자용 인덱스 재생성, 문서 검수 화면

### 5.2 문서 탐색 구조
- 문서 컬렉션
  - 운영가이드, API 문서, 아키텍처, 튜토리얼
- 카테고리
  - 문서 상단 front matter 또는 파일 경로로 결정
- 태그
  - front matter 또는 파일명 규칙으로 생성
- 정렬
  - 최신순, 인기순, 제목순

### 5.3 내비게이션 원칙
- 랜딩에서 허브로 1클릭 진입 가능해야 한다.
- 허브에서 문서 상세로 1클릭 진입 가능해야 한다.
- 상세 페이지에서 같은 카테고리 문서와 인접 문서로 이동 가능해야 한다.
- 모바일에서는 탐색 메뉴, 목차, 공유 버튼이 스크롤을 방해하지 않아야 한다.

## 6. 데이터 모델 설계

### 6.1 문서 메타데이터 모델
```json
{
  "slug": "system-design",
  "title": "SATTIE 시스템 설계 문서",
  "summary": "서비스 구조와 API 계층을 설명하는 설계 문서",
  "source_path": "docs/SYSTEM_DESIGN.md",
  "public_url": "/p/system-design",
  "category": "architecture",
  "tags": ["fastapi", "api", "system-design"],
  "is_public": true,
  "updated_at": "2025-03-14T00:00:00Z",
  "word_count": 3200,
  "reading_minutes": 13,
  "headings": [
    {"level": 2, "text": "시스템 목표와 범위", "anchor": "system-goals"}
  ]
}
```

### 6.2 색인 파일
- `docs_index/documents.json`
  - 전체 문서 메타데이터 목록
- `docs_index/search.json`
  - 제목, 요약, 본문 일부를 포함한 검색용 인덱스

### 6.3 slug 규칙
- 기본값은 파일명 기반 소문자 slug
- 공백은 `-`로 변환
- 한글 제목은 허용 가능하나 URL 안정성을 위해 ASCII slug 별도 생성 권장
- slug 충돌 시 숫자 suffix 추가

### 6.4 front matter 규약
```yaml
---
title: "문서 제목"
summary: "문서 한 줄 요약"
category: "architecture"
tags: ["fastapi", "markdown"]
is_public: true
order: 10
---
```
- `title`
  - 없으면 첫 번째 H1 또는 파일명으로 대체
- `summary`
  - 없으면 본문 첫 문단에서 자동 추출
- `category`
  - 없으면 경로 규칙 또는 `general`로 대체
- `tags`
  - 없으면 빈 배열
- `is_public`
  - 없으면 `true`
- `order`
  - 같은 카테고리 안에서 정렬 우선순위로 사용

### 6.5 문서 상태 모델
- `draft`
  - 업로드되었지만 공개 전 검토 상태
- `published`
  - 허브와 검색에 노출되는 상태
- `hidden`
  - 직접 URL 접근만 허용하거나 목록 비노출 상태
- MVP에서는 내부적으로 `is_public` 불리언으로 단순화하되, 이후 상태 모델로 확장 가능하게 설계한다.
- MVP 운영 화면에서는 상태를 수정하지 않고, `is_public` 값을 읽기 전용으로 표시만 한다.

## 8. API 및 라우트 설계

### 8.1 HTML 라우트
- `GET /`
  - 랜딩 페이지
- `GET /hub`
  - 허브 메인
- `GET /p/{slug}`
  - 문서 상세
- `GET /upload`
  - 업로드 페이지

### 8.2 JSON API
- `GET /api/docs/list`
  - 문서 목록, 카테고리, 태그, 정렬 지원
- `GET /api/docs/search?q=`
  - 제목/요약/본문 검색
- `GET /api/docs/{slug}`
  - 문서 메타데이터 및 본문 HTML
- `GET /api/docs/{slug}/raw`
  - 원본 Markdown
- `POST /api/docs/upload`
  - Markdown 업로드
- `POST /api/docs/reindex`
  - 인덱스 재생성

### 8.3 기존 라우트와의 관계
- `/guide`
  - 초기 운영/검수용으로 잔존 가능
- `/guide/upload`
  - 내부 전환 기간 동안 유지 가능
- `/ui`
  - 위성영상 데모로 유지

### 8.4 에러 응답 원칙
- HTML 페이지 요청
  - 404: 문서를 찾을 수 없음
  - 400: 잘못된 slug 또는 파라미터
  - 500: 렌더링 실패
- JSON API 요청
  - `{"ok": false, "error_code": "...", "message": "..."}`
- 에러 코드 예시
  - `DOC_NOT_FOUND`
  - `INVALID_UPLOAD_TYPE`
  - `INDEX_BUILD_FAILED`
  - `RENDER_FAILED`

### 8.5 업로드 후 처리 흐름
1. 파일 확장자와 이름 검증
2. 저장 경로 확정
3. Markdown 저장
4. 메타 추출
5. 인덱스 갱신
6. 상세 URL 반환
7. 실패 시 저장 롤백 또는 재색인 경고 기록

### 8.6 운영자 접근 원칙
- `/admin/docs`와 재색인 API는 일반 공개 사용자 기능으로 간주하지 않는다.
- MVP에서는 정식 로그인 기능을 넣지 않되, 최소한 환경변수 기반 보호 또는 내부망 전용 운영 중 하나를 선택한다.
- 이 보호 방식은 구현 시작 전에 `3-개발자/` 작업 문서에 고정해야 한다.

## 9. 검색 설계

### 9.1 MVP 검색 방식
- 서버 시작 시 또는 업로드 후 JSON 인덱스 재생성
- 단순 문자열 포함 검색
- 제목 가중치 > 태그 가중치 > 요약 가중치 > 본문 가중치
- `popular` 정렬은 MVP에서 조회수 집계 대신 `order` 오름차순, 동률 시 최신순으로 단순화한다.

### 9.2 검색 대상
- title
- summary
- headings
- tags
- 본문 일부 평문

### 9.3 이후 확장
- SQLite FTS 또는 Meilisearch 연동
- 추천 검색어
- 오타 허용 검색

### 9.4 인덱싱 정책
- 전체 재색인은 서버 시작 시 1회 수행 가능해야 한다.
- 단건 업로드 후에는 증분 재색인을 우선한다.
- 인덱스 생성 실패 시 기존 인덱스를 유지해 읽기 서비스가 중단되지 않게 한다.
- 인덱스 파일 포맷은 사람이 읽을 수 있는 JSON으로 유지한다.

### 11.2 운영자 기능

- 전체 재색인

### 11.3 콘텐츠 시드 운영

## 12. 보안 및 안정성 설계

### 12.1 필수 보안 요구사항
- path traversal 차단
- 업로드 파일 확장자 제한
- HTML 삽입 관련 XSS 점검
- 외부 링크는 `noopener noreferrer` 유지

### 12.2 렌더링 안전성
- 현재 렌더러는 직접 HTML 문자열을 조립하므로 escape 규칙을 엄격히 유지해야 한다.
- front matter, 제목, 요약, heading anchor 생성 과정에서 추가 sanitize가 필요하다.

### 12.3 운영 안정성
- 인덱스 파일이 손상되더라도 원본 Markdown 스캔으로 재구성 가능해야 한다.
- 렌더링 오류가 나는 문서는 목록에서는 보이되 상세에서 오류 문구를 제공한다.

## 13. 성능 설계

### 13.1 MVP 성능 원칙
- 목록 페이지는 인덱스 JSON만 읽는다.
- 상세 페이지는 요청 시 렌더링하되, 캐시 가능하도록 설계한다.
- 업로드 후 필요한 문서만 선택 재색인 가능하게 한다.

### 13.2 캐시 전략
- 공개 문서는 `ETag` 또는 `Last-Modified` 활용
- 인덱스 파일은 메모리 캐시 가능
- Mermaid/MathJax 외부 스크립트 로딩 실패 대비 fallback 유지

## 14. 구현 단계 설계

### 14.1 1단계

- 인덱서 추가

- 팀 워크스페이스

  - 기존 `/guide`와 `/ui`가 깨지지 않음

## 15. 테스트 관점 설계

### 15.1 기능 테스트
- 문서 업로드 후 상세 페이지 접근 가능 여부
- slug 충돌 처리
- Mermaid 문서 렌더링
- MathJax 문서 렌더링
- 이미지 포함 문서 렌더링
- 검색 결과 정렬

### 15.2 보안 테스트
- 파일명 우회 업로드 시도
- 경로 우회 접근 시도
- 스크립트 삽입 텍스트 렌더링 확인

### 15.3 회귀 테스트
- 기존 `/guide` 동작 유지 여부
- 기존 `/ui`와 위성영상 API 동작 영향 여부

### 15.4 수용 테스트 시나리오
- 사용자가 `.md` 파일을 업로드하면 5초 이내에 상세 페이지로 이동할 수 있어야 한다.
- 사용자가 허브 검색창에서 키워드를 입력하면 관련 문서가 우선순위에 맞게 노출되어야 한다.
- 사용자가 문서 상세에서 헤더 링크를 복사해 특정 섹션으로 직접 접근할 수 있어야 한다.
- 사용자가 모바일 화면에서 문서를 열어도 본문, 목차, 공유 기능을 사용할 수 있어야 한다.

## 16. 리스크와 대응

### 16.1 주요 리스크
- 문서 서비스와 기존 위성영상 데모가 하나의 앱에 묶여 있어 관심사가 충돌할 수 있다.
- 직접 구현한 Markdown 렌더러의 확장 시 예외 케이스가 늘어날 수 있다.
- 검색 기능을 뒤늦게 붙이면 URL/메타 구조 재수정이 발생할 수 있다.

### 16.2 대응 방안
- 문서 서비스 모듈을 별도 패키지로 분리
- 메타 추출, 렌더링, 라우팅을 분리해 테스트 가능성 확보
- MVP부터 slug, category, summary, heading anchor를 표준화

## 17. 출시 판단 기준
- 운영자가 샘플 문서 5개 이상을 허브에 올릴 수 있어야 한다.
- 문서 검색, 상세 이동, 업로드, 공유 메타 확인까지 핵심 흐름이 막힘 없이 동작해야 한다.
- 404, 렌더링 오류, 잘못된 업로드에 대한 기본 오류 처리가 사용자에게 이해 가능해야 한다.
- 모바일과 데스크톱에서 주요 문서가 정상적으로 읽혀야 한다.
- 기존 위성영상 데모 기능에 회귀가 없어야 한다.

## 18. 결론
- 현재 프로젝트는 마크다운 뷰어 서비스의 핵심 엔진을 이미 갖고 있다.
- 부족한 것은 렌더링 기능이 아니라 서비스 계층이다.
- 따라서 구현 우선순위는 다음으로 고정한다.
  - 문서 서비스 전용 라우트 분리
  - 메타데이터 인덱스 추가
  - 허브/상세 UI 개선
  - 검색/공유/SEO 기능 추가
- 이 설계는 기존 코드 재사용률을 높이면서도, 이후 독립 문서 플랫폼으로 확장할 수 있는 최소 구조를 제공한다.
