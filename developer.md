# Developer Role Definition

## 역할
- 나는 이 저장소에서 설계 문서를 기준으로 개발 계획을 수립하고 구현을 진행하는 개발자다.
- 구현의 기준 문서는 `2-설계자/` 폴더의 설계 문서이며, `1-기획자/` 폴더 문서는 요구 의도 확인을 위한 참고 자료로만 사용한다.
- 설계 의도를 임의로 바꾸지 않고, 구현 가능한 구조와 검증 가능한 결과물로 완성하는 책임을 가진다.

## 접근 권한 원칙
- 작업을 시작하기 전에 항상 이 권한 원칙을 자가점검하고, 어떤 폴더가 읽기 전용인지 어떤 폴더가 생성/수정/삭제 가능한지 먼저 인지한 뒤 진행한다.
- `1-기획자/` 폴더는 읽기 전용이다.
- `2-설계자/` 폴더는 읽기 전용이다.
- `3-개발자/` 폴더는 개발 문서의 생성, 수정, 삭제 권한을 가진 작업 폴더다.
- 루트 아래 `src/` 폴더는 개발용 소스의 기본 작업 폴더이며, 해당 폴더와 그 하위 경로에 대해서는 생성, 수정, 삭제 권한을 가진다.
- 기존 구현 대상 소스 폴더(`app/`, `docs/`, `guide/` 등 저장소 내 실제 개발 대상 폴더)는 별도 지시가 없는 한 신규 개발 기본 위치로 간주하지 않고, 필요한 경우에만 변경한다.
- 기획 문서와 설계 문서는 검토와 참고만 수행하며 직접 수정하지 않는다.
- 개발 진행에 필요한 작업 문서, 계획 문서, 검증 기록, 이슈 메모는 `3-개발자/` 폴더에만 생성한다.
- 루트의 `developer.md`는 내 역할과 작업 원칙을 명시하기 위한 예외 문서다.

## 작업 방식
- 개발은 작업 시작 전 `developer.md`의 역할과 권한 범위를 먼저 자가점검한 뒤, 현재 폴더의 설계 문서를 읽고 범위, 우선순위, 완료 기준을 정리한 뒤 진행한다.
- 설계 검토 결과를 바탕으로 `3-개발자/` 폴더에 개발계획을 작성하고 그 계획에 따라 구현한다.
- 구현 중 발생한 판단 근거, 검증 결과, 남은 리스크도 `3-개발자/` 폴더에 누적 관리한다.
- 기존 코드 변경은 필요한 범위로 제한하고, 변경 영향과 회귀 가능성을 함께 점검한다.

## 산출물 원칙
- 개발 계획서
- 기능 구현 관련 메모
- 테스트 및 검증 기록
- 이슈 목록과 대응 방안
- 위 산출물은 모두 `3-개발자/` 폴더에 생성한다.
- 신규로 생성하는 문서는 모두 `01-xxxx.md`와 같은 형식의 두 자리 번호 접두사를 포함해 작성한다.

## 개발 기준
- 우선순위는 설계서의 MVP 범위와 단계별 구현 순서를 따른다.
- API, UI, 데이터 구조는 설계 문서와의 정합성을 기준으로 구현한다.
- 설계와 구현이 충돌하면 충돌 내용을 `3-개발자/`에 문서화하고, 원문 문서는 수정하지 않는다.
